## 왜 별도 프로젝트인가 (요약)

PulseLine은 자격증명을 다루지 않는 읽기 전용 상태표시줄이고, QuotaSwitch는 OAuth 토큰을 저장·전환하는 상시 실행 프록시다. 위험 등급과 아키텍처가 근본적으로 달라 한 플러그인에 합치면 저위험 도구가 고위험 공격 표적이 된다 — 이게 사용자가 선택한 "별도 프로젝트로 분리" 방향의 핵심 근거다.

## 핵심 기술 제약 (반드시 숙지)

Claude Code는 `ANTHROPIC_BASE_URL`을 프로세스 시작 시점에만 읽고 세션 내내 고정한다(공식 문서·GitHub 이슈 #8500으로 확인된 사실). 그래서 QuotaSwitch는 "설치 → 계정 등록 → **1회 재시작** → 이후 완전 자동"이라는 흐름을 피할 수 없다. 이걸 숨기지 않고 설정 마법사가 명확히 안내하는 게 Phase 1의 필수 요구사항이다.

## 2026-07-04 (2차) "일간"→"현재 세션" 재확인 + 결함 수정

사용자가 표현을 "일간"에서 "현재 세션 사용량"으로 정정. teamclaude를 재조사해 실제로 **"session (5h) quota"·"weekly (7d) quota"**라는 정확한 용어를 확인 — 즉 처음부터 `QuotaState`에 있던 공식 5h/7d 필드가 사용자가 원한 것이었고, 별도 로컬 집계(`UsageHistory`)로 오버엔지니어링했던 걸 바로잡음. `UsageHistory`는 MVP에서 제외하고 Phase 3 부가 기능(`--history`)으로 격하.

같은 재점검 과정에서 실제 결함도 하나 발견: **`RotationEvent`는 계정 전환이 일어날 때만 생성되는데, 계정이 1개뿐이거나 전환이 한 번도 없으면 "마지막 사용 프로젝트"를 기록할 이벤트 자체가 없어 항상 비어있는 문제**가 있었다. `Account.last_project_path`/`last_used_at`을 신설해 `quotaswitch` 실행마다(전환 여부와 무관하게) 매번 갱신하도록 수정.

## 2026-07-04 (1차) 옵션 설정 + 계정별 사용량 + 프로젝트 경로 추가

세 가지 요청을 반영: ① `quotaswitch config`(전환 임계값·전략·포트를 쉽고 안전하게 조정, claude-swap의 `best`/`next-available` 전략 벤치마킹), ② 계정별 사용량 조회(아래 2차 갱신에서 "세션/주간"으로 재확인됨), ③ 계정별 마지막 사용 프로젝트 경로 표시(CLI 실행 시점 cwd를 그대로 사용해 "최대한 빠르게" 요구 충족, Display 모듈을 거치지 않아 모듈 격리 원칙 유지).

## 2026-07-04 배포 방식 전환: 마켓플레이스 플러그인 → 독립 CLI (가장 큰 변경)

사용자 요청("CLI가 더 편하고 쉽고 강력하고 안전하고 문제없으면 전환")에 따라 teamclaude를 재조사한 결과, `eval $(teamclaude env) claude` 패턴(환경변수를 Claude Code 실행 **전**에 셸에서 설정)을 확인했다. 이 구조를 채택하면 지금까지 마켓플레이스 방식에서 계속 미해결로 남아있던 두 핵심 리스크가 원천 제거된다:
1. **"설정 후 1회 재시작 필요"** — CLI가 Claude Code를 실행하기 전에 환경변수를 이미 설정하므로 재시작 개념 자체가 없어짐
2. **"로컬 프록시 접근 토큰을 Claude Code가 지원하는지 불확실"** — CLI가 요청 경로 전체를 직접 통제하므로 더 이상 가정에 의존하지 않음

**결정**: 독립 CLI(`npm install -g` 또는 curl/PowerShell 스크립트)를 1순위 배포 채널로 전환, Claude Code 플러그인 마켓플레이스는 3순위 "선택적 보조 채널"(CLI를 호출하는 얇은 래퍼)로 격하. "설치·관리할 프로그램 1개"라는 원 요구사항은 오히려 CLI 바이너리 하나로 더 잘 지켜짐. 포기한 것은 `/plugin install` 한 줄의 마켓플레이스 발견성뿐(정직하게 명시).

## 2026-07-04 계정 관리(CRUD) 명령어 신설

teamclaude·claude-swap을 계정관리 관점에서 비교한 결과, 둘 다 "추가·조회·삭제"는 있지만 **라벨 수정 기능이 없고**, teamclaude는 **"삭제가 자격증명까지 완전히 지우는지 문서상 불명확"**하다는 갭을 확인. `/quotaswitch:accounts`(목록, claude-swap `--list` 벤치마킹), `/quotaswitch:remove`(삭제 확인+완전삭제 검증), `/quotaswitch:rename`(신규, 경쟁 도구에 없음), `/quotaswitch:purge`(claude-swap `--purge` 벤치마킹) 4개 명령을 추가하고, 모든 삭제 작업에 확인 절차 + 삭제 후 검증을 Must Have로 못박았다(01_PRD.md, 04_PROJECT_SPEC.md).

## 2026-07-04 포트 충돌 자동 처리로 정책 변경

사용자 요청으로 "포트 충돌 시 단순 종료"(teamclaude 방식)를 폐기하고 **자동 재시도+포트 변경**으로 재결정. 최초 설정 중 충돌은 순차 자동 탐색(최대 10회)으로 완전 자동화. 단, 이미 운영 중인 프록시가 나중에 포트를 다시 못 잡는 경우는 `ANTHROPIC_BASE_URL`이 프로세스 시작 시에만 읽히는 구조적 제약 때문에 자동 전환해도 "재시작이 한 번 더 필요"하다는 한계가 있어 이를 숨기지 않고 문서화했다(04_PROJECT_SPEC.md "포트 충돌 처리" 섹션 신설).

## 2026-07-04 teamclaude 재검토로 4가지 반영

teamclaude 설정 파일(`~/.config/teamclaude.json`)을 실측한 결과 **OAuth 토큰을 평문으로 저장한다는 게 확인됨** — Account 모듈이 OS 자격증명 저장소를 고집하는 이유가 가설이 아니라 증명된 사실이 됨(04_PROJECT_SPEC.md). 그 외: ① 로컬 프록시 API 키 인증 방식이 실제 검증된 패턴임을 확인(NEEDS CLARIFICATION 근거 보강), ② 포트 충돌 시 "자동 재시도 없이 단순 종료" 정책으로 확정(teamclaude 방식 채택), ③ 재평가 주기(`reevalIntervalMs`)·핫 리로드·caam의 sub-100ms 메커니즘(토큰 파일 교체)을 Phase 2~3에 구체적으로 반영, ④ 계정 그룹화·용도별 라우팅은 teamclaude에도 없는 기능임을 확인해 Out of Scope로 명시(오버엔지니어링 방지).

04_PROJECT_SPEC.md에 라이선스(MIT 제안)·외부 코드 재사용 범위·상표권·개인정보 항목을 추가했고, 가장 중요한 변경은 **"상업적 판매·유료 서비스화·회사 납품을 Out of Scope로 명시"**한 것(01_PRD.md §6에 반영). 근거: 할당량 자동 순환 자체가 이용약관과 충돌할 가능성이 아직 법무 검토 전인데, 이를 상업적으로 확장하면 리스크가 개인 사용 대비 크게 증폭되기 때문 — MVP는 무료·개인 사용 목적으로만 배포하는 게 "가장 안전하고 현실적인 방향"이라 판단.
