## 기술 스택

| 영역 | 선택 | 이유 |
|------|------|------|
| 런타임 | Node.js (SEA 배포) | PulseLine과 동일한 근거 — Claude Code가 2026-01부터 Node.js 없는 네이티브 설치를 권장 경로로 쓰므로, SEA로 빌드해 사용자 PC의 Node.js 설치 여부와 무관하게 동작해야 함 |
| 프록시 서버 | Node.js 내장 `http`/`https` 모듈 | teamclaude가 "의존성 없음, Node.js 내장 모듈만 사용"으로 검증한 패턴을 그대로 계승(강점 유지) |
| 자격증명 저장 | OS 네이티브 저장소 우선(Windows Credential Manager/DPAPI, macOS Keychain, Linux libsecret), 미지원 환경만 암호화 파일 폴백 | aurakit-security.md의 "시크릿 하드코딩·평문 저장 금지" 원칙을 직접 구현. claudeline이 macOS Keychain만 우선 지원하고 Windows/Linux를 파일 폴백으로 둔 것과 달리, 3개 OS 모두 네이티브 저장소를 동등하게 우선 시도 |
| 배포 방식 | **독립 CLI(1순위)**, Claude Code Plugin Marketplace(3순위, 선택적 보조 채널) | 2026-07-04 재검토: teamclaude가 `eval $(teamclaude env) claude` 패턴으로 재시작 문제·인증 헤더 불확실성을 원천 제거한다는 걸 확인 — "마켓플레이스+플러그인" 원 요구사항보다 CLI가 사용자의 새 조건("더 편하고 쉽고 강력하고 안전하고 문제없으면 전환")을 더 잘 만족(01_PRD.md 근거 참고) |
| 백그라운드 프로세스 기동 | SessionStart 훅에서 detached child process spawn | Claude Code 플러그인이 MCP 서버처럼 장기 실행 프로세스를 등록할 수 있다는 공식 구조를 재사용 |
| 감사 로그 저장 | 로컬 JSON Lines 파일(암호화 불필요 — 토큰 값을 담지 않으므로) | RotationEvent는 계정 ID·시각·사유만 담아 민감정보가 아님. 별도 암호화 없이 단순 파일로 충분 |

## 계정 관리(CRUD) 요구사항 (2026-07-04 사용자 요청으로 신설)

> 리서치 근거: teamclaude(`login`/`accounts`/`remove`/TUI `d`)와 claude-swap(`--add-account`/`--list`/`--remove-account`/`--purge`)을 비교한 결과, 두 도구 모두 "추가·조회·삭제"는 있지만 **"수정(라벨 변경)"은 둘 다 없음**을 확인했다(신규 확인 사실). 또한 teamclaude는 "삭제가 자격증명까지 완전히 지우는지"가 문서상 불명확하다는 게 확인됨 — QuotaSwitch는 이 두 지점(수정 기능 부재, 삭제 불확실성)을 개선 지점으로 삼는다.

### 명령어 세트 (Must Have, 2026-07-04 CLI 전환으로 슬래시 명령 → 터미널 명령 갱신)
| 명령어 | 동작 | 벤치마킹 대상 |
|---|---|---|
| `quotaswitch setup` | 계정 추가(OAuth) | teamclaude `login`, claude-swap `--add-account` |
| `quotaswitch accounts` | 목록 조회(라벨·5h/7d 사용률·리셋 시각) | claude-swap `--list`(가장 정보량이 많은 벤치마크) |
| `quotaswitch remove <라벨>` | 개별 삭제 | teamclaude `remove`, claude-swap `--remove-account` |
| `quotaswitch rename <기존> <새>` | 라벨 수정(신규 — 경쟁 도구에 없음) | — |
| `quotaswitch purge` | 전체 초기화 | claude-swap `--purge` |
| `quotaswitch switch` | 수동 강제 전환(이미 계획됨) | claude-swap `--switch-to` |
| `quotaswitch` (또는 `eval $(quotaswitch env) claude`) | Claude Code를 프록시 환경변수가 이미 설정된 상태로 실행 — 재시작 개념 자체가 없는 핵심 진입점 | teamclaude `run`/`env` |
| `quotaswitch config [--threshold N] [--strategy best\|next-available] [--port N]` (신규) | 전환 임계값·전략·포트를 대화형 또는 플래그로 조정 | claude-swap `--switch --strategy` |
| `quotaswitch accounts` (재확인) | 계정별 **세션(5h)·주간(7d) 사용률·리셋 시각**을 기본 표시(공식 API 값, 신규 API 호출 불필요) | teamclaude `session (5h) quota`/`weekly (7d) quota`, claude-swap `--list` |
| `quotaswitch accounts --history` (Phase 3, 격하) | 과거 여러 날의 로컬 집계 참고 리포트(부가 기능) | ccusage 벤치마킹 |

> Claude Code 플러그인이 필요하면 Phase 3의 선택적 마켓플레이스 래퍼(`/quotaswitch:*`)가 이 CLI 명령을 그대로 호출한다(01_PRD.md·ClaudeTower 04_PROJECT_SPEC.md 참고).

### 옵션 설정 요구사항 (2026-07-04 사용자 요청으로 신설)
- **쉽고 빠르게**: `quotaswitch config`를 인자 없이 실행하면 현재 설정값 전체를 표로 보여주고, 바꿀 항목만 대화형으로 선택 — 매번 전체 옵션을 다 외워서 플래그로 칠 필요 없음
- **강력하게**: claude-swap이 실제로 제공하는 전환 전략(`best`=할당량 가장 많이 남은 계정 우선, `next-available`=rate-limit 안 걸린 계정으로 즉시 스킵) 두 가지를 옵션으로 제공(실측 확인된 패턴)
- **안전하게**: 값 검증을 거치지 않은 설정은 저장하지 않는다 — `threshold`는 50~100 범위만 허용(50 미만은 너무 잦은 전환을 유발), `port`는 1024~65535 범위만 허용
- **문제없이**: 설정 변경은 다음 `quotaswitch` 실행부터 반영되며, 현재 이미 실행 중인 Claude Code 세션에는 영향을 주지 않는다는 걸 명확히 안내(포트 충돌 처리와 동일한 원칙 재확인)

### 계정별 사용량 표시 요구사항 (2026-07-04 재확인 — "일간"에서 "현재 세션"으로 사용자 표현 정정 반영)

> **자기 정정**: 이전 턴에서 사용자의 "일간 사용량" 요청을 ccusage 스타일의 로컬 집계(별도 `UsageHistory` 엔티티)로 확대해석했으나, 사용자가 이번에 "현재 세션 사용량"으로 표현을 정정했다. teamclaude를 재조사한 결과 실제로 **"session (5h) quota"·"weekly (7d) quota"**라는 정확한 용어를 확인했다 — 즉 사용자가 원한 건 Anthropic 공식 API가 이미 제공하는 `rate_limits.five_hour`(현재 세션)·`rate_limits.seven_day`(주간)였고, 이건 **처음부터 QuotaState 엔티티에 있던 필드**였다. 새로 만들 필요가 없었던 것을 오버엔지니어링했던 셈이라 바로잡는다.

- **Must Have(MVP, 재확인)**: `quotaswitch accounts`가 계정별로 **"세션(5h) 사용률·리셋 시각"**과 **"주간(7d) 사용률·리셋 시각"**을 기본으로 표시 — teamclaude의 `session (5h) quota`/`weekly (7d) quota` 용어를 그대로 채택해 사용자·경쟁 도구 간 혼선을 없앤다. 데이터는 QuotaState(공식 API 실시간 값)에서 직접 가져오므로 별도 로컬 집계나 추가 API 호출이 필요 없다
- **UsageHistory(로컬 집계 일간 리포트)는 Phase 3으로 격하**(MVP에서 제외) — 이건 공식 세션/주간 수치와 별개로 "과거 여러 날의 사용 패턴을 돌아보고 싶을 때"를 위한 부가 참고 기능일 뿐, 사용자가 실제로 요청한 핵심 니즈가 아니었다는 게 이번 재확인으로 명확해졌다. `--timezone` 옵션 등 세부 사항은 Phase 3 착수 시점에 재검토

### 계정별 프로젝트 경로 요구사항 (2026-07-04 사용자 요청으로 신설, 같은 날 재점검으로 결함 수정)
- **최대한 빠르게 감지**: `quotaswitch` CLI 실행 시점의 `process.cwd()`를 그대로 사용 — 별도 파일시스템 탐색이나 git 명령 실행 없이 즉시 얻을 수 있는 값이므로 지연이 0에 수렴(Display 모듈이 `workspace.current_dir`을 즉시 파싱하는 것과 동일한 설계 원칙)
- **모듈 격리 유지**(신규 발견, 02_DATA_MODEL.md 참고): 이 값은 Display 모듈을 거치지 않고 Account 모듈(CLI 진입점)이 스스로 얻으므로, "Display→Account 접근 금지" 원칙을 깨지 않는다
- **(결함 수정, Must Have)** 계정 전환이 일어날 때만 기록되는 `RotationEvent.project_path`에만 의존하지 않는다 — 전환이 한 번도 없어도(계정이 1개뿐이거나 아직 임계값 미도달) 표시가 비지 않도록, `quotaswitch` 실행마다 `Account.last_project_path`를 매번 갱신한다
- `quotaswitch accounts`에 "마지막 사용 프로젝트" 컬럼으로 표시(전체 경로가 너무 길면 폴더명만 축약, Display 모듈의 `dir_name_only` 표시 정책과 일관성 유지)

### 안전 요구사항 (Must Have)
- **삭제 확인 절차**: `remove`·`purge`는 실행 즉시 지우지 않고 "정말 삭제하시겠습니까? (계정: {라벨})" 확인을 거친다 — 실수로 잘못된 계정을 지우는 사고 방지
- **삭제 완전성 검증**(신규, teamclaude 약점 개선): `remove` 실행 후 OS 자격증명 저장소를 다시 조회해 실제로 항목이 사라졌는지 확인하고, 결과를 사용자에게 명시적으로 알린다("계정이 삭제되었고, 저장된 인증 정보도 함께 제거되었습니다" 또는 실패 시 구체적 원인)
- **슬롯/라벨 충돌 방지**: 새 계정 추가 시 기존 라벨과 중복되면 자동으로 변경하지 않고 사용자에게 확인을 요청(claude-swap의 "슬롯 덮어쓰기 전 확인" 패턴 벤치마킹)
- `purge`는 되돌릴 수 없는 작업이므로 확인 문구에 "이 작업은 취소할 수 없습니다"를 명시

### 계정 상태 동기화 (Must Have — "동기화" 요구사항 명확화)
- **자동 동기화**: `reeval_interval_ms`(기본 5분)마다 모든 등록 계정의 할당량 상태를 백그라운드로 갱신(teamclaude 벤치마킹, 이미 반영됨)
- **수동 강제 동기화**(신규): `/quotaswitch:accounts` 실행 시 캐시된 값이 아니라 API를 직접 재조회하는 옵션을 제공해, "지금 당장 최신 상태를 보고 싶다"는 요구를 충족(teamclaude의 `R` 키 재로드, `status` 명령과 유사한 사상)

## 포트 충돌 처리 (2026-07-04 사용자 요청으로 신설)

> 리서치 근거: Node.js 생태계에서 `EADDRINUSE`(포트 사용 중) 에러를 감지해 포트를 순차 증가시키며 재시도하는 것은 `portfinder` 등으로 이미 검증된 표준 패턴(리서치로 확인된 사실). teamclaude는 이 자동화 없이 단순 종료하지만, 사용자 요청으로 QuotaSwitch는 더 적극적으로 자동 처리한다.

### 상황 1 — 최초 설정(QuickSetup) 중 포트 충돌 (Must Have)
- 기본 포트(41411)가 사용 중이면 `41412`, `41413`... 순으로 최대 10회까지 자동 탐색
- 사용 가능한 포트를 찾으면 `ProxyConfig.port`와 `settings.json`의 `env.ANTHROPIC_BASE_URL`을 그 포트로 함께 기록
- **이 시점은 아직 재시작 전이라 완전히 안전하게 자동화 가능**(Claude Code가 아직 이 값을 읽지 않았으므로)

### 상황 2 — 이미 운영 중이던 프록시가 재시작해야 하는데 포트 충돌 (Must Have)
1. 먼저 같은 포트로 짧은 backoff 재시도(1초 간격, 최대 3회) — 일시적 점유(예: OS가 소켓을 아직 반환 안 함)일 가능성이 높으므로
2. 그래도 실패하면 다음 포트로 자동 전환하고 `ProxyConfig.port` + `settings.json`을 갱신
3. **한계를 숨기지 않고 명시**: `ANTHROPIC_BASE_URL`은 Claude Code 프로세스 시작 시점에만 읽히므로(공식 문서로 확인된 구조적 제약), 이미 실행 중인 Claude Code 세션은 자동 포트 변경 후에도 예전 포트를 계속 바라봐 여전히 실패한다. 이 경우 진단 명령(`/quotaswitch:setup` 재실행 등)이 "포트가 바뀌었으니 재시작하세요"를 안내해야 함 — 즉 자동 포트 변경은 "다음 재시작부터의 정상 작동"을 보장하는 것이지, "지금 세션을 즉시 살리는" 기능이 아니다
4. 최대 20개 포트까지 다 막혀 있으면(비정상적 상황) teamclaude처럼 명확한 에러 메시지 출력 후 종료(무한 재시도 방지)

### 왜 이 설계인가
- 최초 설정 시점은 자동화해도 리스크가 없어 사용자 편의를 최대화하고, 런타임 중 변경은 구조적 제약(ANTHROPIC_BASE_URL 고정) 때문에 "부분적 자동화 + 명확한 안내"로 절충 — 안 되는 걸 된다고 포장하지 않는다

## 보안 요구사항 (OWASP ASVS / Secure by Design 매핑)

> 이 프로젝트는 OAuth 토큰을 저장·전환하는 도구이므로 보안은 선택 항목이 아니라 핵심 요구사항이다. 아래는 ASVS 카테고리별로 실제 적용 여부와 우선순위(Must/Should/Could)를 매긴 것 — 해당 없는 항목도 이유를 명시해 "빠뜨린 게 아니라 검토 후 제외했다"는 걸 분명히 한다.

### 인증/인가 (ASVS V2/V4) — Must Have
- 계정 등록은 Anthropic의 공식 OAuth 흐름만 사용, 자체 로그인 시스템을 만들지 않는다
- **OAuth CSRF 방어**: OAuth 요청 시 `state` 파라미터를 암호학적으로 안전한 난수로 생성하고, 콜백에서 검증 없이는 토큰 교환을 진행하지 않는다(리서치 시 확인되지 않았던 빠진 항목 — 표준 OAuth 보안 요구사항이므로 반드시 포함)
- **로컬 프록시 자체 접근 인증(신규, Should Have)**: `127.0.0.1` 바인딩만으로는 같은 머신에서 실행되는 다른 프로세스나 악성 웹페이지(DNS 리바인딩)의 무단 접근을 완전히 막지 못한다. 프록시 기동 시 로컬 전용 랜덤 토큰을 생성해 Claude Code(설정된 `ANTHROPIC_BASE_URL` 요청)만 알 수 있게 하고, 이 토큰이 없는 요청은 거부한다

### 접근 통제 기준
- 로그인 필요 기능: 계정 등록(`quotaswitch setup`)은 Anthropic OAuth 로그인 필요, 그 외 로컬 조회 기능은 불필요
- 역할 구분: 해당 없음(로컬 단일 사용자 — 팀 공유 계정 풀은 Out of Scope, 01_PRD.md §6 참고)
- 서버/API/DB 접근 통제: 로컬 프록시는 위 "로컬 프록시 자체 접근 인증" 기준 적용, 업스트림(Anthropic API)은 계정별 실제 자격증명으로만 접근

### 입력값 검증 (ASVS V5) — Must Have
- 계정 `label` 등 사용자 입력값은 신뢰하지 않는 입력으로 취급, 셸 메타문자·제어문자를 포함해도 안전하게 저장·표시
- Anthropic API 응답(할당량 헤더 등)은 malformed일 수 있다는 전제로 파싱, 실패 시 해당 계정을 즉시 활성화하지 않고 안전한 기본 상태로 유지

### 인젝션 방지 (ASVS V5.3) — Must Have
- 프록시가 업스트림으로 요청을 전달할 때 헤더 인젝션(CRLF 삽입 등)을 방지 — 헤더 값에 개행 문자가 있으면 거부
- 셸 명령을 직접 구성하는 코드가 있다면(예: OS 자격증명 CLI 호출) `execFile`(인자 배열)만 사용, `exec`(문자열 결합) 금지
- SQL Injection·위험한 파일 업로드: 해당 없음 — DB·파일 업로드 기능 없음(감사 로그는 JSON Lines 파일)

### 비밀정보 관리 (ASVS V6/V9) — Must Have (2026-07-04 재검토로 근거 강화)
- **확인된 사실**: teamclaude는 `~/.config/teamclaude.json`에 `accessToken`·`refreshToken`·프록시 `apiKey`를 **평문 그대로** 저장한다(실측 확인 — 설정 파일 예시에 `"accessToken": "sk-ant-oat01-..."`가 그대로 노출됨). 이 파일이 유출되면(백업 동기화, 실수로 커밋, 악성코드 등) 모든 계정이 즉시 탈취된다 — 이것이 QuotaSwitch가 처음부터 OS 자격증명 저장소를 채택한 이유가 실제 경쟁 도구의 약점으로 구체적으로 증명된 사례
- API 키·토큰·비밀번호·인증서·개인 키·DB 접속 정보·OAuth Secret이 코드/문서/로그/Git 추적 대상에 절대 포함되지 않는다
- 저장소 루트에 `.gitignore`로 로컬 설정 파일·감사 로그 디렉터리를 명시적으로 제외(신규 — 기존 문서에 누락돼 있었음)
- 로그·에러 메시지에 토큰 값이 출력되지 않도록 마스킹 로직을 모든 로그 출력 경로에 강제 적용

### 세션/토큰 관리 (ASVS V3/V6) — Must Have
- OAuth 토큰의 만료 시각을 QuotaState/CredentialRef와 함께 저장하고, 만료 임박 시 자동 갱신(claudeline이 검증한 패턴 벤치마킹) — 만료된 토큰으로 계속 요청을 시도해 계정이 잠기는 것 방지
- 로컬 프록시 접근 토큰(위 "로컬 프록시 자체 접근 인증")은 프록시 프로세스 재시작마다 새로 생성, 디스크에 평문 저장하지 않고 메모리 또는 OS 자격증명 저장소에만 유지
- nonce/state 값은 1회용으로만 사용 후 즉시 폐기

### 보안 설정 (ASVS V14) — Should Have
- 운영 빌드(SEA 바이너리)에는 디버그 로그 기본 OFF, 활성화 시에도 토큰 마스킹은 항상 유지(우회 불가)
- 프록시가 HTTP 서버이므로 `Origin`/`Referer` 헤더를 검사해 브라우저발 요청(웹페이지에서의 DNS 리바인딩 공격)을 차단 — Claude Code CLI가 보내는 요청에는 이 헤더들이 없거나 로컬 프로세스 출처임을 활용
- CORS: 브라우저에 응답을 노출하지 않는 것이 기본 정책(Access-Control-Allow-Origin 미설정)

### 의존성 보안 (ASVS V14.2) — Should Have
- OS 자격증명 저장소 연동 라이브러리는 유지보수 활발도·최근 CVE 이력을 배포 전 확인(02_DATA_MODEL.md·본 문서 NEEDS CLARIFICATION과 연계)
- CI에 `npm audit` 고위험 항목 검사 추가

### DB/스토리지 권한 (ASVS V12) — Must Have
- RotationEvent 감사 로그 파일은 소유자만 읽기 가능한 권한으로 생성(POSIX 0600, Windows ACL) — 토큰 값은 없지만 계정 전환 패턴 자체가 민감할 수 있음
- CredentialRef가 가리키는 OS 자격증명 저장소 항목은 해당 OS의 기본 접근 제어(사용자 계정 단위 격리)를 그대로 신뢰 — 별도 애플리케이션 레벨 암호화를 중복 적용하지 않음(불필요한 복잡성 배제)

## 절대 하지 마 (DO NOT)

- [ ] **OAuth 토큰·API 키를 평문 파일이나 로그에 절대 쓰지 마** — 반드시 OS 자격증명 저장소 사용, 디버그 로그에도 토큰 마스킹 필수
- [ ] 프록시를 `0.0.0.0`이나 외부 인터페이스에 바인딩하지 마 — 항상 `127.0.0.1` 고정
- [ ] `127.0.0.1` 바인딩만으로 접근 통제가 끝났다고 가정하지 마 — 로컬 프록시 접근 토큰 검증 없이 요청을 처리하지 말 것(같은 머신의 다른 프로세스·DNS 리바인딩 공격 대비)
- [ ] OAuth `state` 파라미터 검증 없이 콜백에서 토큰 교환을 진행하지 마
- [ ] 저장소에 `.gitignore` 없이 로컬 설정·감사 로그 디렉터리를 커밋하지 마
- [ ] teamclaude 등 참고 도구의 코드를 그대로 복사하지 마 — 아이디어·패턴 참고만 허용
- [ ] "이용약관 위반 아님을 보장한다"처럼 확인되지 않은 사실을 단정하는 문구를 문서에 쓰지 마 — 항상 "개인 책임 하에 사용", "[법무 검토 필요]"로 표현할 것
- [ ] MVP를 상업적 판매·유료 서비스·회사 납품 목적으로 배포하지 마 (Out of Scope, 01_PRD.md §6)
- [ ] 자격증명 조회·전환 경로에서 에러를 삼키지 마 — 실패 시 명확히 사용자에게 알릴 것(silent fallback 금지, aurakit-security.md와 무관하게 이 도구 특성상 더 엄격히 적용)
- [ ] RotationEvent 로그를 끄거나 생략하지 마 — 이 도구는 사용자 모르게 계정을 바꾸는 도구라 감사 로그 없이는 신뢰할 수 없음
- [ ] 확인 절차 없이 `remove`·`purge`를 즉시 실행하지 마
- [ ] `remove` 실행 후 OS 자격증명 저장소에서 실제로 지워졌는지 검증하지 않고 "삭제 완료"라고 표시하지 마
- [ ] "이용약관 위반 가능성" 고지 없이 배포하지 마 — README 최상단 필수
- [ ] **사용자 동의 없이 셸 프로필(.bashrc/PowerShell profile)을 자동으로 수정하지 마** — 셸 wrapper 모드(Phase 2)는 반드시 명시적 opt-in 후에만 적용, 기존에 사용자가 정의한 `claude` alias/함수가 있는지 먼저 감지하고 충돌 시 적용 중단
- [ ] `--no-verify`, `try/catch`로 보안 검증(자격증명 저장소 접근 실패 등)을 조용히 우회하지 마

### 예상 문제·불편·위험·오류·변수·충돌·실패 목록 (README 트러블슈팅에 반드시 반영, Must Have)
- **`quotaswitch` 대신 평소처럼 `claude`를 직접 실행해서 "왜 전환이 안 되지?"라고 착각**(2026-07-04 CLI 전환으로 재정의 — 기존 "재시작 깜빡함" 리스크가 이 형태로 대체됨): 가장 흔할 것으로 예상되는 혼란 — `quotaswitch status`로 "지금 프록시가 개입 중인지" 바로 확인 가능하게 하고, 이걸 FAQ 최상단에 배치. 이 마찰을 완전히 없애고 싶으면 Phase 3의 영구 셸 별칭 등록(opt-in)으로 `claude` 자체를 wrapper로 만들 수 있음을 안내
- **로컬 프록시 포트(예: 41411)가 다른 프로그램과 충돌**: 최초 설정 시에는 자동으로 다른 포트를 찾아 해결되므로 사용자가 신경 쓸 필요 없음(이 사실을 README에 명시해 불안 해소). 단, 이미 정상 작동 중이던 프록시가 나중에(예: 재부팅 후) 포트를 다시 못 잡으면 자동으로 새 포트를 잡되 **재시작이 한 번 더 필요할 수 있다**는 점을 트러블슈팅에 명확히 안내
- **OAuth 로그인 중 브라우저가 자동으로 안 열림**: 원인(기본 브라우저 미설정, 방화벽 등)과 수동으로 링크 여는 방법
- **같은 계정이 이미 다른 기기·다른 프로그램에서 로그인되어 있어 충돌**: 발생 가능성과 대처
- **OS 자격증명 저장소 접근 실패**: Windows Credential Manager/macOS Keychain/Linux libsecret이 잠겨있거나 권한 문제일 때의 증상과 대처
- **회사 보안 소프트웨어·방화벽이 로컬 프록시를 악성 프로그램으로 오인해 차단**: 로컬 전용(127.0.0.1) 프록시라는 걸 보안팀에 설명할 수 있는 근거 자료 제공
- **VPN 사용 중 로컬 프록시가 우회되거나 충돌**: 확인 안 된 변수 — 정확한 동작은 실측 필요, README에는 "VPN 환경에서는 아직 충분히 검증되지 않았다"고 정직하게 표기(과장 금지 원칙)
- **계정 전환 도중 진행 중이던 작업이 끊기는지**: 실제로는 API 요청 단위 전환이라 세션 자체는 유지되는 것으로 설계되어 있으나, 실사용 검증 전까지는 "이론상"이라는 표현 사용(안티패턴#1 — 검증 안 된 것을 확정처럼 쓰지 않음)
- **가장 심각한 위험 — 이용약관 위반으로 계정 정지 가능성**: 축소·희석 없이 README 최상단에 고지, "이 도구를 쓰다가 계정이 정지되어도 개발자는 책임지지 않는다"는 문구를 보증 없음(AS-IS) 조항과 함께 명시
- **셸 wrapper 모드(Phase 2, opt-in) 적용 시 기존에 사용자가 만들어둔 `claude` 별칭과 충돌**: 적용 전 자동 충돌 감지 결과를 사용자에게 보여주고 진행 여부 확인
- **계정을 삭제(등록 해제)해도 OS 저장소에 토큰이 남아있는 경우**: `/quotaswitch:remove`가 삭제 후 자동으로 검증까지 하므로 정상적으로는 발생하지 않지만, 검증 실패 시 수동 정리 절차를 안내
- **계정을 잘못 삭제했는데 되돌리고 싶은 경우**: 삭제는 되돌릴 수 없다는 것을 명확히(재등록만 가능), 그래서 확인 절차가 왜 중요한지 설명

### 예상 질문(FAQ) 목록 (Must Have, 최소 이 항목들 포함)

- "회사에 납품하거나 팔아도 되나요?" → 안 됨(01_PRD.md §6, 04_PROJECT_SPEC.md 법률 섹션 연동)
- "이 프록시가 인터넷의 다른 사람도 쓸 수 있나요?" → 아니요, 내 컴퓨터에서만(127.0.0.1) 작동

- "실수로 계정을 다 지웠는데 복구되나요?" → 안 됨, 삭제는 되돌릴 수 없어 확인 절차를 반드시 거침

- "이 계정을 어느 프로젝트에서 썼는지 기억나나요?" → 네, `quotaswitch accounts`에 마지막 사용 프로젝트가 표시됨
